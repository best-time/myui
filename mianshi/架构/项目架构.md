一、高级前端做架构，核心就 4 句话（背会）

- 先分层，再模块化
- 先规范，再开发
- 先可扩展，再写业务
- 先监控，再上线
- 架构不是炫技，是让项目：好维护、好扩展、好协作、好排查。

二、企业级前端架构 8 大设计维度（完整体系）

我按真实项目搭建流程一步步讲，你面试直接按这个顺序说。

1. 技术选型架构（最优先）
   高级前端必须做的事：根据业务场景选技术
   - （1）**框架**选型
     中后台系统 → React + TypeScript + Ant Design
     移动端 / H5 → React/Vue + Vite + 自适应方案
     SSR/SEO → Next.js / Nuxt.js
     小程序 → Taro / Uniapp
   - （2）**构建**工具
     开发 → Vite
     生产 → Rollup / Webpack
   - （3）**状态**管理
     简单项目 → Context / Pinia
     中大型项目 → Zustand / Redux Toolkit
     服务端状态 → React Query / SWR
   - （4）**工具链**
     TypeScript 必开
     ESLint + Prettier + Husky + Lint-Staged
     Jest + RTL 测试

我会根据项目规模、团队习惯、业务复杂度、性能要求做技术选型，不盲目追新，保证稳定、高效、可维护。


2. 项目目录架构（标准企业级）

```
   架构师必须规划清晰目录，让所有人按规范写
   plaintext
   /src
   /api           # 接口请求层（统一封装）
   /assets        # 静态资源
   /components    # 公共组件
   /constants     # 常量/枚举
   /hooks         # 全局 hooks
   /layout        # 布局
   /pages         # 页面
   /router        # 路由
   /store         # 状态管理
   /styles        # 全局样式
   /types         # TS 类型
   /utils         # 工具函数
   核心思想：分层架构
   API 层：所有请求统一管理
   组件层：公共业务组件
   Hook 层：逻辑复用
   页面层：只做渲染
   工具层：通用能力
```

3. 权限架构设计（企业项目灵魂）

```
   高级前端必懂权限架构
   标准 3 层权限：
   路由权限：菜单能不能看见
   按钮权限：增删改查能不能点
   数据权限：能看哪些数据
```

实现方案：
```javascript
登录获取 token → 存入 cookie
路由守卫校验登录状态
获取用户角色 / 菜单列表
动态生成路由（addRoute）
按钮级权限指令（v-auth / hasAuth）
接口层面数据过滤
```

一句话总结：
权限从 路由 → 按钮 → 数据 三层控制，实现完整的 RBAC 权限体系。

4. 网络请求架构（统一封装）

- 统一 baseURL
- 请求 / 响应拦截
- token 自动携带
- 401/403 自动跳转登录
- 错误统一提示
- 重复请求拦截
- 接口缓存
- 加载状态管理

```
   业务代码最终只需要这么写：
   js
   const { data } = useRequest('/api/user/list', { params })
```

5. 组件架构设计（高复用、低耦合）

```
   组件设计原则：
   公共组件：和业务无关（Button、Table、Form）
   业务组件：多个页面使用
   页面组件：只在当前页面使用
   原子化 + 组合式
   
```

组件规范：
- 统一 props
- 支持扩展
- 无副作用
- TS 类型完备

6. 状态管理架构（清晰、不混乱）

```
   架构原则：
   服务端状态 → React Query/SWR
   客户端状态 → Zustand/Redux
   表单状态 → 组件内管理
   状态按模块拆分
   禁止大仓库、禁止滥用全局状态
```

7. 工程化 & 规范架构（高级前端标志）
   必须包含：

```
   ESLint + Prettier 代码规范
   Husky + lint-staged 提交校验
   commitlint 提交信息规范
   CHANGELOG 自动生成
   环境配置（dev/test/prod）
   打包优化（gzip、分包、CDN）
   监控体系（错误、性能、请求）
   规范大于技术！
```

8. 性能 & 体验架构（最后一环）
   
   架构层面提前内置优化：
   - 路由懒加载
   - 图片懒加载
   - 虚拟列表
   - 打包分包
   - 接口防抖节流
   - 缓存策略
   - 骨架屏
   - 错误边界
   - 日志监控

三、面试官问：
项目架构会从8 个维度进行体系化设计：

- **技术选型**：根据业务场景、团队情况、项目规模选择框架、构建工具、生态库。
- **目录分层**：采用 API、组件、Hook、页面、工具、状态分层架构，职责清晰。
- **权限体系**：设计路由权限、按钮权限、数据权限三层 RBAC 权限架构。
- **网络封装**：统一请求、拦截、错误处理、缓存、加载状态。
- **组件设计**：原子化 + 业务组件分层，高复用、低耦合、强类型。
- **状态管理**：服务端状态 + 客户端状态分离，模块化管理。
- **工程化规范**：代码规范、提交规范、自动化校验、统一构建部署。
- **性能与监控**：内置性能优化、错误监控、体验优化。

最终目标是让项目**可维护、可扩展、可测试、可监控**，同时提升团队开发效率和线上稳定性。

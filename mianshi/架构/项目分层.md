### 标准5层
页面层 → 组件层 → 业务逻辑层 → 数据服务层 → 基础工具层

依赖规则：只能上层 import 下层，禁止下层引入上层


分层核心目标：解耦、复用、易维护、可测试、业务与技术隔离


1. 第一层：页面层 Views / Pages（最上层，入口）
   职责：路由对应页面，只负责组装组件、绑定路由、接收路由参数、简单页面状态分发
   - 不写复杂业务逻辑、不直接发请求、不操作缓存
   - 仅做：布局拼装、全局弹窗、路由跳转、页面初始化调用 hooks
```   src/pages/
   Order/
   index.vue / index.tsx  # 页面入口
   config.ts              # 页面常量、表头、枚举
   ```
2. 第二层：组件层 Components
   拆两类，职责隔离：

   （1）业务业务组件 biz-components
   和当前业务强绑定，依赖业务枚举、业务类型、接口
   例：订单列表卡片、商品 sku 选择器、退款弹窗

   （2）通用基础组件 components /ui
   纯 UI，无业务，可全项目复用，只依赖工具、样式
   例：表格、弹窗、输入框、按钮、分页、骨架屏
   规则：组件只做渲染，复杂逻辑抽离到 hooks，不直接请求接口

3. 第三层：业务逻辑层 Hooks / Stores（核心分层，解耦 UI 与逻辑）
   承载所有业务计算、状态管理、流程控制，UI 层只调用，不关心实现
   3.1 组合式 Hooks（Vue3 / React）
   单个页面 / 模块独立逻辑，轻量化复用
   ```
   // hooks/useOrderList.ts
   export function useOrderList() {
   const list = ref([])
   // 调用接口层获取数据
   const getList = () => orderApi.getOrder()
   // 业务计算、筛选、格式化
   return { list, getList }
   }
   ```
   页面直接使用，逻辑完全抽离。
   3.2 全局状态 Store（Pinia / Redux / Zustand）
   跨页面共享状态：用户信息、全局字典、全局配置、购物车
   - store 内部调用 api，不写在页面
   - 只存全局数据，页面局部状态放组件 /hook
   
4. 第四层：数据服务层 Service / Api（对接后端）
   唯一一层发起网络请求，隔离接口地址、请求头、参数处理、响应格式化
   细分两块：
   api 接口定义
   按模块拆分，统一管理 url、请求方法、入参出参 TS 类型
   ```
   // api/order.ts
   import request from '@/utils/request'
   export function getOrderList(params) {
   return request.get('/api/order/list', { params })
   }
   ```
   - request 请求封装
   统一拦截器：token、错误提示、超时、loading、统一响应解构
   - Mock 层
   本地开发模拟接口，切换开关不侵入业务代码
   扩展：本地持久化存储也归这一层（localStorage、indexDB 封装）

5. 第五层：基础工具层 Utils / Core（底层，纯技术能力）
   无任何业务代码，全项目通用工具，所有上层均可调用
   分类：
   - utils：格式化（日期、金额）、防抖节流、树处理、正则
   - constants：全局常量、通用枚举
   - types：全局 TS 类型定义
   - core：项目核心能力（路由守卫、权限、全局事件、埋点、全局样式）
   - assets：静态资源图片字体

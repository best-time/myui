1. 架构分层（核心中的核心）
   把项目按职责拆分成独立层，层与层之间解耦。
   API 层
   工具层
   组件层
   业务逻辑层（Hook）
   页面渲染层
   状态管理层
   好处：
   改业务不动底层，换组件不动页面，加功能不影响旧代码。


2. 面向接口 / 抽象编程，不面向实现
   高级前端标志
   定义统一接口，不依赖具体库
   依赖倒置：高层不依赖低层
   换库、换插件、换服务无需改业务代码
   例子：
   封装请求层 → 以后把 axios 换成 fetch，业务代码一行不动。


3. 模块化、组件化、原子化设计
   大功能拆小模块
   公共逻辑抽成 Hook
   公共 UI 抽成组件
   单一职责：一个组件只做一件事
   可扩展表现：
   加功能 = 加组件 / 加 Hook，不改动旧代码。


4. 配置化 > 硬编码
   所有可变的东西全部放入配置：
   菜单
   路由
   按钮权限
   表单 / 表格列
   枚举 / 状态
   好处：
   需求变更 = 改配置，不改逻辑。


5. 低耦合、高内聚
   模块之间只通过接口通信
   不互相依赖、不互相调用内部方法
   避免 “牵一发动全身”


   可扩展表现：
   删除一个功能，不影响其他功能。


6. 可插拔式架构（Plugin System）
   企业级架构必备：
   功能做成插件
   可安装、可卸载、可替换


   例如：
   - 上传插件、图表插件、编辑器插件、支付插件
   - 可扩展表现：
   - 新功能即插即用，不侵入主体架构。


7. 统一规范 & 设计系统
   统一组件规范
   统一 API 规范
   统一状态规范
   统一命名规范
   统一错误处理规范
   规范越统一，扩展越轻松。


8. 依赖隔离，方便技术升级 / 替换
   第三方库统一封装
   不直接暴露在业务中
   想换库只改封装层


   例子：
   封装 useRequest → 以后把 React-Query 换 SWR，业务完全不动。


9. 预留扩展点（架构师意识）
   真正高级的架构，会提前预留：
   钩子（Hooks）
   插槽（Slots）
   回调（Callbacks）
   中间件（Middleware）
   未来加逻辑，只需要挂载扩展点，不修改核心代码。

三、面试官问：
你如何保证前端项目的可扩展性？
可扩展性主要从架构设计、代码规范、抽象能力、扩展预留四个方向落地：

- 架构分层解耦：将项目分为 API 层、Hook 层、组件层、页面层、状态层，每层职责单一、依赖稳定。
- 面向抽象编程：统一封装请求、组件、工具，业务不依赖具体库，方便未来替换技术栈。
- 模块化 + 组件化 + 配置化：功能模块化、逻辑 Hooks 化、业务配置化，减少硬编码，提高复用性。
- 低耦合高内聚：模块之间通过接口通信，不互相依赖，避免牵一发动全身。
- 可插拔设计：功能插件化，支持即插即用，方便未来业务扩展。
- 制定统一规范：组件、API、命名、错误、状态全部标准化，让扩展可预期。
- 预留扩展点：在核心流程中预留钩子、插槽、中间件，让未来功能扩展无需侵入核心代码。

最终让项目达到：**需求变更不重构、功能扩展不改旧代码、技术升级不影响业务**。

### 一、前端性能优化完整分类（通用手段）
1. 网络层面优化（加载速度）
    - 资源压缩：JS/CSS 代码压缩、Tree-Shaking 剔除无用代码、图片压缩（webp/avif）
    - 资源拆分：路由懒加载、组件按需引入、大文件分包、第三方库拆包（splitChunks）
    - 缓存策略：HTTP 强缓存 Cache-Control、协商缓存 Etag；本地 localStorage 缓存接口数据；静态资源哈希文件名
    - CDN 加速：静态资源托管 CDN，减少服务器带宽、缩短请求距离
    - 请求优化：接口合并、分页 / 分片加载、取消冗余请求、接口节流防抖、预加载 / 预请求
    - 减少体积：移除未使用依赖、替换轻量库（moment → dayjs、lodash → lodash-es）
2. 渲染层面优化（页面卡顿、白屏）
    - 虚拟滚动：万条列表不渲染全部 DOM，只渲染可视区域（react-window/vue-virtual-scroller）
    - 减少重排重绘：避免循环操作 DOM、批量修改样式、使用 transform 代替 top/left、离屏 DOM
    - 大计算剥离：复杂计算放到 WebWorker，不阻塞主线程
    - 图片优化：图片懒加载、占位图、响应式图片、大图分片、禁止超大背景图
    - 骨架屏、Loading、分包首屏：拆分首屏最小关键资源，减少白屏时间
3. 框架代码层优化（Vue/React）
    - 避免无效渲染
    - React：useMemo、useCallback、React.memo，减少组件重复渲染
    - Vue：computed 缓存计算值、v-once、v-memo、避免模板内复杂表达式
    - 合理拆分组件：粗粒度大组件拆细，局部更新不整页刷新
    - 清理副作用：定时器、监听、订阅、事件销毁，防止内存泄漏
    - 状态管理优化：拆分全局状态，局部状态放组件内，不全局存储临时数据
4. 构建工程优化（打包慢、产物大）
    - Vite/Webpack 构建缓存、多线程编译
    - 第三方依赖预构建、外部化 CDN（externals），不打包大型库
    - 关闭开发环境冗余插件，生产环境移除调试工具、console
    - 按需引入 UI 组件库，不全局引入全部组件
5. 运行时体验优化
   - 防抖节流：搜索输入、窗口缩放、滚动事件
   - 长列表虚拟滚动、分页、后端分页，不一次性拉取全量数据
   - 预加载下一页、缓存表格筛选条件，重复打开无需重新请求
   - 大文件分片上传、断点续传

### 二、真实线上优化案例（中后台百万级数据列表页面，Vue3 + Vite + Element Plus）
1. 原始问题
    - 企业用户管理后台，用户列表页面存在严重性能问题：
   - 页面打开白屏 2～3s，首屏加载慢；
   - 后台用户数据量最大 8000 条，一次性全量返回，页面渲染卡顿，滚动掉帧；
   - 搜索、筛选、切换分页时，页面卡死 1s 以上；
   - 打包产物主包 1.8MB，部署后静态资源加载慢；
   - 多处定时器、弹窗监听未销毁，长时间打开页面内存持续上涨。
2. 分步优化手段（落地可验证）
   （1）接口 & 数据优化（最见效）
    - 后端改为分页查询，禁止一次性返回 8000 条，每页 50 条；
   - 高频筛选接口增加 30s 本地缓存，相同条件不再重复请求；
   - 接口返回字段裁剪，只返回页面展示字段，剔除大量无用扩展字段，接口响应体积减少 60%。
   （2）长列表渲染优化（解决滚动卡顿）
   - 引入 vue-virtual-scroller 虚拟滚动，只渲染可视区域 20 条 DOM，原本 8000 条 DOM 直接降到几十条，滚动丝滑无掉帧。
   （3）工程打包优化，减小首屏包体积
   - UI 库改为按需引入，删除全局全量导入 Element Plus；
   - 大型第三方库 xlsx、echarts 使用 externals 外部 CDN 引入，不打入主包；
   - 开启 Vite 构建压缩、Tree-Shaking，删除项目未使用组件与工具函数；
   - 静态图片全部转 WebP，压缩图片体积。
   - 优化后主包从 1.8MB 降至 620KB，首屏加载速度提升一倍。
   （4）Vue 代码渲染优化
   - 表格行渲染使用 v-memo，行数据不变则不重新渲染行；
   - 模板内复杂计算抽离为 computed 缓存，避免每次渲染重复计算；
   - 搜索输入框添加防抖 300ms，减少频繁接口请求；
   - 弹窗、详情组件使用 v-if 而非 v-show，销毁多余 DOM；
   - 统一封装组件销毁钩子，清除定时器、滚动监听、接口订阅，解决内存泄漏。
   （5）加载体验优化，解决白屏
   - 页面增加骨架屏，接口请求期间展示骨架，消除空白视觉等待；
   - 路由组件分包懒加载，非首页路由单独打包；
   - 表格增加加载态、空数据兜底，避免用户无反馈。
   （6）缓存策略优化
   - 静态资源添加哈希文件名，配置 CDN 强缓存；
   - 表格筛选条件、分页页码本地缓存，重新进入页面恢复上次查询条件，无需重复筛选。
3. 优化前后指标对比
    - 首屏白屏时间：2.7s → 0.8s
   - 列表滚动帧率：20 帧左右 → 稳定 60 帧
   - 分页 / 筛选响应卡顿：1.2s → 0.2s
   - 主包体积：1.8MB → 620KB
   - 页面长时间打开内存涨幅：持续上涨 → 稳定无泄漏
4. 落地踩坑总结
    - 一开始只做前端渲染优化，没改后端全量返回接口，性能提升极小，数据层优化优先级最高；
   - 虚拟滚动需要固定行高，动态高度需要额外适配，否则会出现滚动错位；
   - 大量使用 v-memo、useMemo 不能滥用，缓存本身有开销，简单数据无需缓存；
   - 内存泄漏最难排查，必须统一规范组件销毁逻辑，定期用 Chrome Performance 面板检测。


   面试一句话总结
   前端性能优化分为网络、渲染、框架代码、工程构建、运行体验五大类；
   我真实处理过中后台万级用户列表卡顿案例，通过后端分页、虚拟滚动、分包拆包、缓存、清理副作用等组合方案，
   大幅提升首屏速度与滚动流畅度，其中接口数据裁剪与虚拟滚动是收益最高的核心优化手段。
